# 恢复说明

## 两种恢复方式

### 方式 A：本地恢复（推荐新手首选）

适合：

- 手里已经有 `.tar.gz` 归档
- 目标机没有远端凭据
- 不想先处理 `rclone` / `OCI` / `R2`

```bash
vpsmagic restore --local /path/to/backup.tar.gz
```

### 方式 B：远端恢复

适合：

- 目标机已具备远端访问条件（至少有 `rclone.conf`）
- 想直接从云端拉取最新备份
- 跨机恢复时减少手工传文件

```bash
vpsmagic restore
```

远端访问前置条件（视为安全验证门槛，而非可选项）：

- 已配置 `rclone.conf`
- 如果远端依赖 `OCI`，已准备 `/root/.oci/config`

## 恢复前先跑一次 doctor

```bash
vpsmagic doctor
```

重点关注 4 个字段：

| 字段 | 含义 |
|------|------|
| `当前建议` | 最终推荐操作 |
| `风险等级` | 低 / 中 / 高 |
| `阻塞项` | 必须先解决，否则不应正式恢复（如缺 rclone、缺 OCI 凭据） |
| `注意项` | 不是立即阻塞，但恢复后需要更多人工确认（如 Systemd、数据库） |

## 恢复流程

### 远端恢复流程（共 7 步）

```
1. 查找可用备份（本地优先 → 输入 0 切换云端）
2. 下载备份文件 (rclone copy)
3. SHA256 校验
4. 解密（如启用了加密）
5. 解压归档
6. 确认恢复范围（需输入精确 yes）
7. 按模块顺序执行恢复 + 健康检查
```

### 本地恢复流程（共 5 步）

```
1. SHA256 校验
2. 解密（如启用了加密）
3. 解压归档
4. 确认恢复范围（需输入精确 yes）
5. 按模块顺序执行恢复 + 健康检查
```

## 恢复模块执行顺序

恢复模块按以下固定顺序执行（先恢复基础设施，再恢复依赖它们的服务）：

| 顺序 | 模块 | 恢复内容 |
|------|------|---------|
| 1 | Crontab | 用户 crontab、系统 cron 目录、`/etc/crontab` |
| 2 | 自定义路径 | 按路径映射还原文件和目录 |
| 3 | 防火墙 | iptables/ip6tables/UFW/nftables/firewalld/fail2ban（**自动保留当前 SSH 端口**） |
| 4 | Docker Compose | 项目文件、卷数据、bind mounts、权限修复、`docker compose up -d` |
| 5 | 独立 Docker | 容器元数据（**需手动重建**） |
| 6 | 反向代理 | Nginx/Caddy/Apache 配置、自动安装/启动/reload |
| 7 | 数据库 | SQLite 自动恢复；MySQL/PostgreSQL **仅提示手动导入命令** |
| 8 | SSL 证书 | Let's Encrypt、acme.sh 证书目录 |
| 9 | Systemd 服务 | .service 文件、工作目录、程序目录、**自动重建 Python venv + pip install** |
| 10 | 用户目录 | dotfiles、rclone.conf 等（**自动跳过 .ssh/authorized_keys 保护当前 SSH 访问**） |

## 自动安装能力

恢复时如果目标机缺少关键依赖，restore 会自动尝试安装：

| 依赖 | 安装方式 |
|------|---------|
| Docker + Compose | APT 软件源 → Docker 官方仓库（自动添加 GPG key + apt source） |
| Caddy | APT 软件源 → Caddy 官方仓库（自动添加） |
| Nginx / Apache | 系统包管理器 |
| rclone | 官方 release zip 下载 → 系统包管理器 |
| python3-venv | APT 安装（用于重建 Systemd 服务的 Python 虚拟环境） |

> **注意**：自动安装仅支持 Debian/Ubuntu (APT)。其他发行版可能需要手动安装。

## 关键安全机制

### 防火墙恢复后保留 SSH

恢复防火墙规则后，restore 会自动检测当前 SSH 监听端口，确保这些端口在 iptables/ip6tables/UFW 中保持放行，**不会因恢复源机防火墙规则而丢失当前 SSH 连接**。

### 恢复前配置级快照

恢复前会自动创建轻量级快照，覆盖范围：

- `/etc/caddy`、`/etc/nginx`、`/etc/apache2`
- `/etc/systemd/system`
- `/etc/ufw`、`/etc/cron.d`、cron 定时目录
- 当前 iptables/ip6tables/nftables 规则
- root 用户的 crontab
- Compose 项目的 compose 文件和 .env

快照目录位于：`/opt/vpsmagic/backups/restore/snapshots/`

### 用户目录保护

恢复 `user_home` 时，会**跳过覆盖 `.ssh/authorized_keys`**，确保当前 SSH 密钥认证不受影响。

### Caddy TLS 自愈

恢复 Caddy 配置后，如果检测到 TLS 状态异常（如 JWS 验证失败），会自动清理本机 Caddy TLS 缓存并重启服务，等待证书重新签发。

## 健康检查（恢复后自动运行）

恢复完成后自动执行以下检查：

| 检查项 | 判断条件 |
|-------|---------|
| Docker Compose 服务 | running 服务数 = 预期服务数 |
| Compose 端口 | 所有 published 端口均在监听 |
| Compose 出网 | 容器内 HTTPS 探测 cloudflare.com |
| Systemd 服务 | `systemctl is-active` 状态 |
| 反向代理 | caddy/nginx/apache2 active 或监听 80/443 |
| 代理端口 | 80、443 是否监听 |
| rclone 可用性 | 可用 remote 数量 |

## 单实例服务保护

以下服务恢复后**不会自动启动**（避免新旧机同时运行冲突）：

- `status.env` 中标记 `START_POLICY=manual` 的服务
- 明确命中 Telegram bot 相关特征的服务

这些服务需要在 DNS 切换完成后手动启动：

```bash
systemctl start <service-name>
```

## 恢复后检查清单

1. 摘要里 `错误: 0`
2. Docker Compose：容器数量正确、关键端口监听、出网 ok
3. 反向代理：caddy/nginx 状态 active、80/443 监听
4. 单实例服务：确认按预期"已恢复但未自动启动"
5. 数据库：MySQL/PostgreSQL 需手动导入 SQL 文件
6. DNS：A 记录指向新 IP
7. SSL：证书签发完成

> 如果恢复后网站打不开，先分层排查：服务 → 反代 → 端口 → 证书 → DNS

## 无人值守恢复

```bash
vpsmagic restore --auto-confirm --rollback-on-failure
```

- `--auto-confirm`：跳过所有交互确认
- `--rollback-on-failure`：检测到关键恢复步骤失败时自动执行轻量回滚
- 回滚范围：**仅配置级内容**，不回滚卷数据、数据库结果或业务副作用
