# 业务画像与适用场景

这篇文档不是按 `VPSA` 或某一台具体机器写的，而是按**常见 VPS 业务形态**来判断：

- 你的机器更像哪一类
- 适合先走哪种备份 / 恢复路径
- 哪些能力现在可以依赖
- 哪些能力仍应保守看待

## 最先做的事

先执行：

```bash
bash vpsmagic.sh doctor
```

这个命令会输出：

- 当前机器画像
- 检测到的业务形态
- 依赖与远端条件
- 恢复等级建议
- 推荐起步路径

如果你还没有任何配置，也可以直接先跑这个命令。

## 常见业务画像

### 画像 A：轻量通用 VPS

典型特征：

- 没有或几乎没有 Docker 业务
- 只有少量目录、Crontab、反代、简单服务
- 你只是想先把当前机器状态备份下来

建议路径：

1. 先用 `vpsmagic init`
2. 选 `仅本地备份`
3. 先做本地备份
4. 再做本地恢复演练

最适合先验证：

- 本地归档
- `sha256` 校验
- 本地恢复流程

### 画像 B：标准 Compose 应用型

典型特征：

- 主要业务跑在 Docker Compose
- 有 Caddy / Nginx 之类标准反代
- 自定义路径有限
- 独立容器不多或没有

这是当前支持最成熟的一类。

建议路径：

1. 先本地备份
2. 先本地恢复演练
3. 再配置远端
4. 再做跨机恢复

这类机器当前通常可把这些模块作为主恢复目标：

- Docker Compose
- 反向代理
- Crontab
- 防火墙

### 画像 C：带数据库的 Compose 应用型

典型特征：

- Compose 是主业务形态
- 同时有 MySQL / PostgreSQL / SQLite
- 业务恢复不仅要看服务是否起来，还要看数据一致性

建议路径：

1. 先做本地备份
2. 先做恢复演练
3. 重点验证数据库和业务数据是否合理
4. 再谈云端恢复或正式切换

注意：

- 数据库文件或导出可以恢复
- 但这不等于业务一致性天然恢复

### 画像 D：Systemd 服务型

典型特征：

- 业务主要靠 `/etc/systemd/system/*.service`
- 不一定有 Docker
- 常见于 bot、守护进程、脚本服务

建议路径：

1. 先看 `doctor` 检测到多少自定义 Systemd 服务
2. 先本地备份
3. 恢复后重点核对服务是否应该自动启动
4. 单实例服务要单独决定切换时机

### 画像 E：混合 Docker 业务型

典型特征：

- 既有 Compose，也有独立容器
- 目录结构和运行方式不统一
- 自定义挂载、外部依赖、手工改动较多

这是风险更高的一类。

建议路径：

1. 不要直接假设“自动恢复一定完整”
2. 先把 Compose 部分当主恢复目标
3. 把独立容器视为“重建线索”
4. 正式切换前必须做恢复演练

## 如何理解恢复等级

### A 级

表示当前项目对这类模块的自动恢复支持较成熟。

典型包括：

- Docker Compose
- 标准反向代理
- Crontab
- 防火墙

### B 级

表示能恢复到一个较可用的状态，但恢复后必须人工确认。

典型包括：

- 常见 Systemd 服务
- 用户目录关键配置
- 数据库导出与文件恢复

### C 级

表示当前工具更像是在帮你“保留线索”，而不是承诺自动拉起。

典型包括：

- 独立 Docker 容器
- 强依赖目标机凭据但尚未预置的远端场景
- 业务层副作用回滚

## 什么时候应该先停止继续自动化

如果你遇到下面任一情况，建议先不要继续堆自动化操作：

1. 远端前置依赖不满足
2. `sha256` 校验失败
3. 恢复摘要里出现错误
4. 服务虽然起来了，但业务访问仍不对
5. 这台机器的业务形态你自己都还没说清楚

## 对新客户机器的实用建议

面对一台陌生 VPS，最稳的顺序是：

1. 先运行 `vpsmagic doctor`
2. 先判断机器属于哪类画像
3. 先做一次本地备份
4. 先做一次本地恢复演练
5. 再配置远端
6. 再做跨机恢复

不要把“我们之前实验过的机器”当成默认模板。  
你真正要信的是：

- 这台机器被识别成了什么画像
- 当前模块分别属于 A / B / C 哪一级
- 哪些结果已经被演练验证过
