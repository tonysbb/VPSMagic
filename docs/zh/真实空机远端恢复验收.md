# 真实空机远端恢复验收

## 验收目标

本次验收模拟的是一台真实空机目标机，而不是沿用我们联调时已经准备好的环境。

目标是确认以下链路是否成立：

1. 新用户可以通过 `init` 生成远端恢复配置
2. `doctor` 能按配置和语言输出接入建议
3. `restore` 能在远端前置不完整时明确报缺失项
4. `OCI` 不可用时能回退到其他已配置远端
5. Debian 空机上缺少 `docker-compose-plugin` 时，工具能继续尝试 Docker 官方仓库
6. 最终恢复结果通过健康检查，摘要为 `警告: 0`、`错误: 0`

## 本次验收环境

- 源机：`NCPDE`
- 目标机：`WAWHK`
- 备份来源：远端对象存储
- 配置路径：`/opt/vpsmagic/config.env`
- 恢复命令：

```bash
bash vpsmagic.sh restore --config /opt/vpsmagic/config.env --source-hostname NCPDE
```

## 已验证通过

- `init` 可生成远端恢复配置
- `doctor --config` 会读取 `UI_LANG`
- `restore` 在已配置远端时，会直接进入远端恢复前置检查
- `OCI` 缺 `/root/.oci/config` 时，会明确提示原因
- `R2` 可作为可用远端继续恢复
- 本地已有同名归档且校验一致时，会直接复用
- 恢复前会创建配置级快照
- Debian 默认源缺 `docker-compose-plugin` 时，会继续尝试 Docker 官方仓库
- `downloader`、`pdfmaker`、`caddy`、`downnow-bot` 这轮都恢复成功
- 最终摘要为：
  - `警告: 0`
  - `错误: 0`

## 本次确认下来的边界

### 1. `rclone.conf` 是安全验证门槛

工具不会自动生成远端访问配置。

这不是缺陷，而是刻意保留的安全边界：

- 你需要明确知道目标机将访问哪些远端
- 你需要明确知道使用哪个 `rclone remote`
- 你需要自行决定是否把远端访问能力授予目标机

所以远端恢复前，目标机仍应由使用者主动准备：

- `/root/.config/rclone/rclone.conf`

### 2. `OCI` 凭据也是安全验证门槛

对于 `OCI` 主目标，工具不会自动生成或复制：

- `/root/.oci/config`
- 对应私钥材料

这同样是刻意保留的安全边界。

如果目标机没有 `OCI` 凭据：

- 工具会明确提示
- 并在有其他已配置远端时回退到备选远端
- 无法满足前置条件时，仍可改走 `restore --local`

### 3. 轻量回滚仍然只是配置级

这次验收没有改变这个边界：

- 不回滚卷数据
- 不回滚数据库结果
- 不回滚业务副作用

## 对普通使用者的实际建议

1. 先用 `doctor` 判断业务画像和接入路径
2. 先完成本地备份 + 本地恢复演练
3. 再配置远端恢复
4. 把 `rclone.conf`、`OCI` 凭据视为上线前的安全验证门槛
5. 首次正式切换前，至少完成一次目标机恢复演练
