# 排障说明

## 先判断你卡在哪一层

大多数恢复问题可以先分成 4 类：

1. 备份文件本身就不存在或校验失败
2. 远端访问条件不满足
3. 恢复动作执行了，但服务没起来
4. 服务起来了，但业务仍不可用

先判断属于哪一类，再继续排障，效率会高很多。

一个经验原则：

- 不要一开始就同时怀疑“备份坏了、远端坏了、挂载坏了、token 坏了、证书坏了”
- 先把问题压缩到一层，再继续往下查

## 远端恢复前置失败

常见原因：

- 缺少 `rclone.conf`
- OCI 缺少 `/root/.oci/config`
- 目标机无法访问远端存储
- Debian 默认软件源只有 `docker.io`，但没有 `docker-compose-plugin`

建议：

1. 优先看“远端恢复前置检查”输出
2. 凭据无法补齐时，先用 `restore --local`
3. 不要把“远端不可用”误判成“备份根本不存在”

如果前置检查里已经明确说某个远端不可用，后面的重点就不是“它有没有备份”，而是“为什么当前机器不具备访问条件”。

如果问题出在 `Docker / Compose`：

- 恢复过程中如果发现目标机没有 `Docker`，工具会先尝试自动安装
- 在 Debian 系统上，如果默认源只有 `docker.io`、没有 `docker-compose-plugin`，工具会继续尝试添加 Docker 官方仓库
- 如果这一步仍失败，恢复摘要里会明确给出失败原因，便于你手动处理

## SHA256 校验失败

先检查：

- 本地归档是否完整
- `.sha256` 是否和归档来自同一轮备份
- 远端下载是否中断或被覆盖

如果校验失败，不建议继续恢复。

先解决校验问题，再讨论恢复问题。

## Compose 服务恢复后不通

优先检查：

- 容器是否全部 `running`
- 端口是否监听
- Compose 出网是否 `ok`
- 反向代理是否 `active`

如果宿主机能上网、容器不能上网，优先怀疑 Docker bridge / 防火墙，不要先怀疑挂载或 token。

这是一个高频误区。很多“挂载全失效”“上传链路异常”最后根因其实是容器出网断了。

## 恢复成功但网站打不开

优先区分：

1. 服务根本没起来
2. 服务起来了，但反代没起来
3. 反代起来了，但证书还没签发完成
4. 证书正常，但 DNS / Cloudflare 还没切对

## Cloudflare / Caddy 证书问题

优先看：

```bash
journalctl -u caddy -n 100 --no-pager
```

如果刚切 DNS，短时间内出现 `525` 并不一定是最终故障，应先确认 ACME 挑战是否已成功。
